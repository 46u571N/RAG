§ 01 — Motivación

¿Por qué necesitamos RAG?

Los LLMs son poderosos, pero tienen tres limitaciones críticas que RAG ataca directamente.

📅
Límite de conocimiento

Todo modelo tiene una fecha de corte de entrenamiento. Información posterior —o datos privados— le resulta completamente invisible.

RAG conecta el modelo a una base actualizable en tiempo real, sin reentrenar nada.
💭
Alucinaciones

Cuando un LLM no sabe algo, a veces lo inventa con total confianza. En contextos clínicos, legales o corporativos esto es inaceptable.

RAG reduce las alucinaciones: el modelo genera solo a partir de documentos reales, con fuente citable.
💸
Costo del reentrenamiento

Reentrenar un modelo grande cuesta millones de dólares y semanas de cómputo. Actualizar políticas semanalmente es inviable.

Actualizar una base vectorial cuesta centavos y segundos. El modelo no se toca.
§ 02 — Arquitectura

El pipeline RAG en 5 etapas

Hacé clic en cada paso para expandir los detalles del flujo.

①
Pregunta del usuario
Consulta en lenguaje natural
›

El sistema recibe la consulta tal como la escribe el usuario. No requiere sintaxis especial ni palabras clave exactas — el lenguaje natural es suficiente.

ej: "¿Cuántos días de vacaciones tengo con 6 años en la empresa?"
②
Vectorización
Texto → embedding numérico
›

La pregunta se pasa por un modelo de embeddings que la transforma en un vector de alta dimensión. Ese vector captura el significado semántico, no solo las palabras literales.

pregunta → [0.3, 0.8, -0.14, 0.67, ... ×1536 dims]
③
Recuperación
Top-K fragmentos por similitud coseno
›

El vector de la pregunta se compara contra todos los embeddings pre-calculados de los documentos en la base vectorial. Se devuelven los K fragmentos más cercanos geométricamente.

similitud_coseno(q_vec, chunk_vec) → ordenados desc → top-3
④
Aumentación del contexto
Pregunta + fragmentos → prompt enriquecido
›

Los fragmentos recuperados se insertan junto a la pregunta original en un prompt estructurado. El LLM recibe instrucciones, contexto con fuentes y la consulta del usuario.

[sistema] + [chunks con scores] + [pregunta] + [guardrail]
⑤
Generación con LLM
Respuesta fundamentada y citable
›

El LLM genera la respuesta condicionado por el contexto provisto — no por su memoria de entrenamiento. Puede citar fuentes, admitir incertidumbre, y producir texto coherente.

output: respuesta + fuente_doc + página + similitud
§ 03 — Representación semántica

¿Qué es un embedding?

Un embedding convierte texto en un punto en el espacio matemático. La distancia entre puntos refleja similitud de significado. Seleccioná una palabra para ver cómo se relaciona con las demás.

Seleccioná:
Seleccioná una palabra para ver sus similitudes.
La propiedad mágica: aritmética semántica

Los embeddings capturan relaciones conceptuales de forma aritmética:

vector("rey") − vector("hombre") + vector("mujer") ≈ vector("reina")

§ 12 — Decisión práctica

¿Cuándo y qué embebés?

El mismo stack técnico (BGE-M3 + pgvector) sirve para tres patrones distintos. Elegir el incorrecto es el error más común en implementaciones reales. Tres preguntas para llegar al patrón correcto para tu caso.

Inicio
¿Qué querés embeber?

Esta es la pregunta más importante. Define todo lo demás: qué modelo usás, dónde corre, y qué problema resolvés.

Inicio›Preguntas del usuario
¿Las preguntas pueden contener datos privados?

Por ejemplo: "¿cuál es el saldo de mi cuenta?" incluye contexto del cliente. "¿cuál es el horario de atención?" no incluye nada privado.

Inicio›Datos del negocio
¿Los datos tienen texto libre o son principalmente numéricos?

La búsqueda semántica aporta valor cuando hay lenguaje natural que interpretar. Para números, fechas y códigos estructurados, SQL exacto siempre gana.

Inicio›Datos del negocio›Texto libre
¿Los datos contienen información personal o confidencial?

Registros médicos, datos de pacientes, información de clientes identificables. Si el dato puede identificar a una persona, es privado.

Inicio›Preguntas›Con datos privados
🏠 Embedding local obligatorio
Caché semántico — BGE-M3 local

Embebés las preguntas del usuario, no los datos del negocio. Los datos contables o clínicos quedan en su base original intactos. Solo guardás pares (embedding de la pregunta → respuesta ya generada) en pgvector.

BGE-M3 debe correr en tu infra local: si mandás las preguntas a una API externa para embeber, esas preguntas (que contienen datos del cliente) salen de tu infraestructura. El mismo problema que usar Gemini API para esto.

🏠 Embedding local obligatorio
RAG sobre datos — BGE-M3 local

Acá sí embebés los datos del negocio: cada observación clínica, nota médica, o campo de texto libre se convierte en un embedding y se indexa en pgvector. Cuando el usuario pregunta, buscás por similitud semántica sobre esos embeddings.

Esto permite búsquedas que SQL no puede hacer: "pacientes con signos compatibles con infección" encuentra registros que dicen "leucocitosis", "proceso febril agudo" o "cuadro compatible con bacteremia" — aunque ninguno diga la palabra exacta buscada.

BGE-M3 local es obligatorio: mandar texto clínico a una API externa significa que esos datos del paciente salen de tu infra. Incompatible con cualquier regulación de salud.

Inicio›Datos›Numérico / estructurado
🗃️ SQL — no uses embeddings sobre estos datos
SQL exacto — sin capa semántica sobre los datos

La búsqueda semántica aporta valor cuando hay lenguaje natural que interpretar. Para datos contables, financieros o de inventario — saldos, fechas, montos, códigos — SQL exacto siempre gana: más preciso, más rápido, sin ambigüedad.

Intentar buscar "pérdidas" semánticamente en una base contable puede asociarlo con "gastos de papelería" o "mermas de inventario". En auditoría esa imprecisión es un problema grave.
